# Interim Project Results Summary

This project focuses on large language model training on Huawei Ascend 910A NPUs. Its goal is to enable key training capabilities under the constraints of older hardware and lower CANN versions, using a `PyTorch + torch_npu` stack. These capabilities include tensor parallelism (TP), data parallelism (DDP), pipeline parallelism (PP), MoE model adaptation, and real-data training workflows. At the current stage, the project has moved beyond basic training validation and entered a phase of engineering adaptation for specific models on a constrained platform.

Based on the current repository state, the project has already established a custom tensor-parallel training foundation for Ascend NPUs. This includes column-parallel and row-parallel linear layers, a custom backward-compatible `all_gather` implementation, HCCL-oriented communication handling, and automatic HuggingFace model conversion into TP-compatible structures. The value of this work is not merely in implementing parallel training in theory, but in providing a practical and runnable parallelization path under `torch_npu` and HCCL.

On the model side, the project has further added dedicated support for the DeepSeek family, especially `DeepSeek-V2-Lite`, together with tensor-parallel support for MoE architectures. The most important outcome here is not simply “MoE support” in the abstract, but the engineering adaptations made for the limitations of the older NPU software stack. For example, the project replaces problematic NPU operations such as `torch.nonzero` and `.any()` with fixed-shape matrix-based computation and batch processing logic. These changes materially improve runnability and stability on Ascend 910A and represent one of the most platform-specific and hardest-to-replace parts of the current work.

On the training workflow side, the project has already gone beyond synthetic data or toy examples. It now supports direct loading of offline HuggingFace Arrow datasets, real-data training scripts, long-run training configurations, validation perplexity calculation, checkpoint saving, and checkpoint-based resume. This indicates that the project has progressed from “verifying parallel capabilities” to “executing realistic training tasks,” which gives the current work substantial engineering value.

When compared with recent public releases of Huawei’s official MindSpore / MindSpore Transformers stack, one point is clear: the official ecosystem has already added a broad set of general-purpose capabilities, including TP, PP, sequence parallelism, MoE training, DeepSeek mainline model support, weight conversion, dataset handling, memory optimization, and fault recovery. Therefore, from a pure capability-definition perspective, part of this project’s work—especially generic parallel training infrastructure, baseline MoE support, and general model integration—no longer represents an entirely uncovered space.

However, this does not mean that the current implementation lacks value. The key issue is that Huawei’s publicly documented new training stack primarily targets newer A2/A3 platforms and higher CANN versions, whereas our actual environment is `Ascend 910A + lower CANN + PyTorch/torch_npu`. Under this constraint, the fact that the official framework “has a capability” does not mean that it can directly replace what is currently running in our environment. In other words, the official stack now covers many general capabilities in a newer software and hardware ecosystem, while this project provides a practical and deployable solution for an older platform.

For that reason, the project’s achievements should be described more precisely. The current value should not be framed mainly as “inventing a brand-new parallel training framework from scratch,” nor simply as “implementing TP/MoE capabilities that the official ecosystem does not have.” A more accurate description is that this project has successfully enabled the critical training path for large models on 910A and low-CANN environments using the existing `PyTorch + torch_npu` stack. In particular, it has produced a runnable, reproducible, and extensible adaptation layer for DeepSeek-V2-Lite, MoE models, HCCL communication compatibility, NPU operator workarounds, and real-data training.

From this perspective, the project’s current-stage achievements can be summarized in four points. First, it provides a tensor-parallel training foundation tailored to Ascend 910A. Second, it adds dedicated adaptation for DeepSeek-V2-Lite and MoE architectures under constrained environments. Third, it resolves a number of NPU-specific compatibility and stability issues. Fourth, it establishes an end-to-end real-data training, validation, and checkpoint-resume workflow. Together, these form the core practical value of the project at this stage.

Looking ahead, if the project continues, the main effort should no longer be placed on re-implementing general parallel capabilities for their own sake. Instead, the more valuable directions are: first, improving the stability, efficiency, and training ceiling of the current legacy-platform solution, including communication optimization, memory optimization, and broader model-structure coverage; and second, clarifying the boundary between what can eventually migrate to the official stack and what must remain as local adaptation assets for the 910A generation. This will help define a clearer technical roadmap and a more accurate narrative for the project’s outcomes.
