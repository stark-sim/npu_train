# 后续工作建议清单

## 目标

在 `Ascend 910A + 低版本 CANN + PyTorch/torch_npu` 约束下，继续沿着当前优化方向推进，但优先选择那些 **对现有环境真正有价值**、且 **不只是重复官方 MindSpore 新框架通用能力** 的工作。

## 建议继续做

### P1. TP Checkpoint 合并 / 重分片 / 导出工具

**建议理由：**
当前 TP 训练的保存方式仍然比较原始，`examples/train_tp_custom.py` 里还是按 rank 单独保存模型权重。这会直接影响以下能力：
- 恢复训练的便利性
- 不同 `tp_size` 间迁移
- 导出为单份 HuggingFace 权重
- 和后续评测、部署脚本对接

**价值判断：**
这属于当前技术栈上的工程基础设施，实用价值非常高，也不属于 MindSpore 已经替代你们的重点。

**建议产出：**
- `merge_tp_checkpoint.py`
- `reshard_tp_checkpoint.py`
- `export_hf_from_tp.py`
- 配套格式说明文档

---

### P2. NPU 兼容性层系统化

**建议理由：**
目前仓库里已经存在针对 NPU 不兼容算子的定点绕行，例如避免 `torch.nonzero`、`.any()` 等操作。这类 work-around 很有价值，但现在更多是散落在具体实现里。

**价值判断：**
下一步最值得做的不是继续零散修补，而是把这些经验抽象成“能力探测 + fallback 策略 + 已知限制矩阵”的兼容层。这会显著提高后续维护效率，也更符合 910A 老环境的真实需求。

**建议产出：**
- 一份 NPU 能力探测模块
- 一份算子兼容性矩阵
- 通用 fallback helper
- 常见错误码与规避策略文档

---

### P3. 910A 专用 Profiling / Autotune 工具

**建议理由：**
当前仓库虽然已有 benchmark 脚本，但还没有形成“自动搜索最优训练配方”的能力。对 910A 这种受限环境来说，能快速找到稳定可跑的参数组合，比继续扩展通用框架更有现实意义。

**价值判断：**
这是典型的平台工程资产，不是官方新框架现成替代物。

**建议产出：**
- 自动扫描 `batch_size / seq_len / tp_size / dtype`
- 自动记录 tokens/sec、显存、失败类型
- 输出推荐配置文件或最佳实践报告

---

### P4. 长时间训练的稳定性与自动恢复

**建议理由：**
你们已经有 checkpoint 保存与恢复基础，但对 910A + 低 CANN 这种环境，真正影响训练效率的往往不是单步速度，而是长跑过程中的中断、OOM、HCCL timeout 和异常退出。

**价值判断：**
如果把“自动恢复、失败重试、训练续跑”做好，这会比继续堆通用优化更有长期收益。

**建议产出：**
- watchdog / launcher wrapper
- OOM/HCCL timeout 自动重试
- 自动接续最近 checkpoint
- 失败日志汇总工具

## 有条件再做

### P5. 混合 TP+DDP 的正确性与组管理完善

**现状：**
代码里已经明确写到混合 TP+DDP 的 process group 管理还不完整，目前更多是按纯 TP 假设实现。

**建议判断：**
如果后续训练确实要走 TP × DP，这项工作就值得补；否则可以放在第二优先级。

---

### P6. MoE 通信路径优化

**现状：**
当前 MoE router 仍偏复制式实现，dispatch/gather 逻辑也是“先保证正确，再谈最优”的写法。

**建议判断：**
这项优化有潜力，但前提是你们已经验证当前瓶颈确实主要落在 MoE 通信，而不是模型加载、编译预热、显存约束或长时间稳定性上。

## 建议暂缓

以下方向目前不建议继续投入太多时间：

- 更通用的新并行框架抽象
- 通用 sequence parallel / context parallel 扩展
- 重做一套更完整的 MoE 框架设计
- 仅为了“对齐官方能力”而补的功能
- 在没有明确瓶颈证据前，直接做完整 MLA attention TP

这些方向要么与官方新框架能力高度重合，要么实现成本很高，但对你们当前环境的边际收益有限。

## 推荐优先级

1. **P1：TP Checkpoint 合并 / 重分片 / 导出**
2. **P2：NPU 兼容性层系统化**
3. **P3：910A Profiling / Autotune**
4. **P4：长时间训练稳定性与自动恢复**
5. **P5：混合 TP+DDP 正确性完善**
6. **P6：MoE 通信优化**

## 总结

如果继续沿当前路线推进，最有意义的方向不是重复建设“官方已经有的通用能力”，而是沉淀 **老平台可运行、可恢复、可迁移、可调优** 的工程资产。换句话说，下一阶段最值得投入的，不是“再造一个更大的训练框架”，而是把当前这套 910A 方案打磨成真正可长期使用的基础设施。
